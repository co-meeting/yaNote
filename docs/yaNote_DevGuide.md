# yaNote Generative AI 開発ガイド

このガイドは、yaNote をコーディングエージェント（Claude Code 等）とどのように開発しているかをまとめたものです。  
yaNote の全成果物は生成AIとの対話で作成・管理されており（[yaNote Manifesto](./yaNote-Manifesto.md) 参照）、このガイドはその開発方法の技術解説にあたります。

---

## 1. 目的

- 生成AIとの協業による yaNote の機能実装およびドキュメント生成を、一貫性のある品質で行うための指針を示す。
- 開発方法の変遷と現在の開発体制を記録し、同様の開発スタイルを試みる人の参考とする。

---

## 2. 開発方法の変遷

yaNote の開発方法は、生成AIツールの進化に合わせて変化してきました。

1. **チャットAI時代（初期〜）**  
   ChatGPT / Claude のプロジェクト機能にソースコードとドキュメントをアップロードし、対話で仕様を検討。修正箇所をコードとして出力してもらい、手動でソースに適用していた。コード量が増えるにつれ全体の再生成は現実的でなくなり、修正箇所のみの手動適用が中心になった。

2. **エージェント過渡期**  
   仕様のまとめまではチャットAIで行い、実際のコード修正は Cline などの初期のコーディングエージェントに依頼するスタイルに移行。「出力されたコードをどう安全に適用するか」という問題がツール側で解決され始めた。

3. **コーディングエージェント時代（現在）**  
   Claude Code 等のコーディングエージェントがリポジトリを直接読み書きするため、ファイルのアップロードや修正の手動適用は不要になった。仕様の議論から実装・レビュー・ドキュメント更新までを、リポジトリ上での対話として一貫して行っている。リポジトリの規約やリリース手順は `CLAUDE.md` に集約し、エージェントが常に参照できるようにしている。

この変遷に伴い、かつてソースコード冒頭に置いていた【DO NOT MODIFY】開発ガイドラインコメント（全文再生成時に規約が失われるのを防ぐためのもの）は役割を終え、廃止しました。コーディング規約は `CLAUDE.md` に記載しています。

---

## 3. 開発プロセス

1. **仕様の議論**  
   - 追加・修正したい機能について、その仕様や実装方法をエージェントと議論する。  
   - エージェントはリポジトリの現状（コード・仕様書・help.md）を直接参照できるため、既存動作との整合性を踏まえた議論ができる。

2. **実装**  
   - 議論がまとまったら、エージェントに実装を依頼する。エージェントがソースコードを直接修正する。

3. **コードレビュー**  
   - 修正されたコードをレビューし、意図どおり実装されていることを確認する。エージェント自身のレビューに加え、別セッションや別のAIによるレビューを行うこともある。

4. **動作テスト**  
   - ブラウザでの手動テストで機能が正しく動作するか確認する。確認すべき操作は [yaNote 操作仕様書](./yaNote-OperationSpec.md) とタッチ操作仕様書に定義されている。  
   - 問題が見つかった場合はエージェントと議論し、解決するまでステップ1から反復する。

5. **ドキュメント更新**  
   - 機能開発が完了したら、各種ドキュメント（仕様書、help.md、README.md）とリリースノートをエージェントに作成・更新してもらう。  
   - 操作に影響しない修正の場合はリリースノートのみで終了することもある。

6. **リリース**  
   - バージョン番号の同期箇所を含むリリース手順は `CLAUDE.md` に記載。全て完了したら GitHub の main にコミットし、リリースする（main へのコミット＝リリース）。  
   - バージョンアップはなるべく細かく行う。

---

## 4. エージェントとのコミュニケーション

### 4.1 明確な要件提示
- 各機能や修正点について、具体的な要件と動作仕様を記述すること。  
  例：「Cmd/Ctrl+A による全オブジェクト選択機能を実装したい。テキスト編集中はブラウザの全選択動作もさせたいが、どう思う？」
- 仕様が固まっていない段階では、実装を依頼する前に議論・提案を求める形で対話する。

### 4.2 恒久的な指針は CLAUDE.md に残す
- コーディング規約、リリース手順、アーキテクチャの説明など、セッションをまたいで有効な指針は `CLAUDE.md` に記録する。  
- 対話の中で決めたルールが今後も有効なら、その場で `CLAUDE.md` への反映まで依頼する。

---

## 5. ドキュメント運用

- ソースコード以外のドキュメントはすべて Markdown 形式で、既存の文書と同一の文体・粒度の日本語で記述する。
- 機能変更時は仕様書（yaNote-OperationSpec.md）、help.md、README.md の整合性を保つ。生成されたドキュメントに不備があれば、機能開発と同様に対話で反復して改善する。

---

このガイドに沿ってコーディングエージェントと協業することで、yaNote プロジェクトの開発が円滑に進み、各種ドキュメントおよびソースコードが一貫性のある品質で維持されることを期待します。
